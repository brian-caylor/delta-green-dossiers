# Beta Test & Code Audit — 2026-03-15

> Comprehensive review of Delta Green Agent Dossier app covering mechanical fidelity,
> UI/UX, code quality, edge cases, and feature gaps. Each finding is tagged with
> severity, affected files, and a concrete action so a coding agent can plan and
> implement without ambiguity.

---

## How to use this document

- **BUG** — Incorrect behavior relative to Delta Green rules or app intent.
- **FRICTION** — Works but causes unnecessary player pain during live sessions.
- **ENHANCEMENT** — Missing feature that would materially improve the experience.
- **DEBT** — Code-quality issue that doesn't affect users today but blocks future work.
- **MINOR** — Small inconsistency or hygiene issue.

Priority: P0 = do first, P1 = do next, P2 = when convenient.

### 1. Auto-derive HP, WP from base stats

| | |
|---|---|
| **Type** | BUG |
| **Priority** | P0 |
| **Files** | `src/components/tabs/StatsTab.jsx` (lines 18-23), `src/data/defaultCharacter.js` (line 16) |
| **Current behavior** | Changing STR, CON, or POW only updates the stat score. Derived max values (HP, WP) remain unchanged. The player must manually edit HP max and WP max after every stat change. |
| **Expected behavior (rules)** | HP max = `round((STR + CON) / 2)`. WP max = POW score. These should auto-recalculate whenever STR, CON, or POW change. |
| **Why it matters** | Stat damage is common in Delta Green (adaptation costs 1d6 CHA or POW, injuries reduce STR/CON). Forgetting to update derived stats is the #1 error on paper sheets — the app should eliminate this, not replicate it. |

**Implementation guidance:**

In `StatsTab.jsx`, inside `updateStatScore`, after updating the stat score, recompute
and write derived max values:

```jsx
// After the existing stat update in updateStatScore:
updateChar(c => {
  const updated = { ...c, stats: { ...c.stats, [key]: { ...c.stats[key], score: v } } };
  const str = Number(updated.stats.str.score) || 0;
  const con = Number(updated.stats.con.score) || 0;
  const pow = Number(updated.stats.pow.score) || 0;

  const newHpMax = Math.round((str + con) / 2);
  const newWpMax = pow;

  return {
    ...updated,
    derived: {
      ...updated.derived,
      hp: { ...updated.derived.hp, max: newHpMax },
      wp: { ...updated.derived.wp, max: newWpMax },
    },
  };
});
```

**Edge cases to handle:**
- If current HP > new max, clamp current to new max.
- If current WP > new max, clamp current to new max.
- Log the max change to sessionLog (e.g., `HP max 10→8`).
- On PDF/JSON import, recalculate derived max from imported stats to correct
  any stale values.

### 2. Combat quick-reference panel

| | |
|---|---|
| **Type** | FRICTION |
| **Priority** | P0 |
| **Files** | New component needed; data sourced from `activeChar.derived`, `activeChar.skills`, `activeChar.weapons` |
| **Problem** | During combat — the highest-tension moment — a player needs HP, Dodge%, Firearms%, Melee Weapons%, and their weapon damage. These are currently spread across 3 tabs (Stats, Skills, Combat). Tab-switching mid-firefight is slow and error-prone. |

**Implementation guidance:**

Create a collapsible "Combat Quick Ref" strip that appears at the top of the tab
content area (or as a floating bar) when the Combat tab is active, or optionally on
all tabs. It should display:

```
HP: 8/10  |  WP: 6/10  |  SAN: 34/50  |  Dodge: 50%  |  Firearms: 40%  |  Unarmed: 60%  |  Melee: 30%
```

- Pull skill values by name from `activeChar.skills` for Dodge, Firearms,
  Melee Weapons, and Unarmed Combat.
- Make the skill list configurable (let the user pin/unpin skills).
- Keep it compact: a single horizontal bar, 28-32px tall, monospace font.

### 3. Inline dice roller on skills

| | |
|---|---|
| **Type** | ENHANCEMENT |
| **Priority** | P1 |
| **Files** | `src/components/tabs/SkillsTab.jsx`, new utility `src/utils/diceRoller.js` |
| **Problem** | Every skill shows a percentage but there's no way to roll against it in-app. Players must context-switch to a separate dice app or physical dice, then mentally compare the result. |

**Implementation guidance:**

Add a small dice icon button next to each skill value in `SkillsTab.jsx`. On click,
roll d100 and display a brief inline result.

Dice logic utility:

```js
// src/utils/diceRoller.js
export function rollD100(target) {
  const roll = Math.floor(Math.random() * 100) + 1;
  const pass = roll <= target;
  // Criticals: matched digits (11, 22, 33...) that are also successes
  const isCritical = roll === 1 || (pass && roll % 11 === 0);
  // Fumbles: matched digits that are failures, or 100
  const isFumble = roll === 100 || (!pass && roll > 5 && roll % 11 === 0);
  return { roll, target, pass, isCritical, isFumble };
}
```

UI behavior:
- Show result inline for ~3 seconds (or until next roll), styled:
  - Green for pass, red for fail, gold for critical, skull-red for fumble.
- Auto-check the "failed" checkbox if the roll fails (since the player just
  demonstrated a failed skill use).
- Optional: add the roll to the session log.

### 4. SAN max not auto-set from POW on character creation

| | |
|---|---|
| **Type** | BUG |
| **Priority** | P1 |
| **Files** | `src/data/defaultCharacter.js` (line 16), same `updateStatScore` fix as finding #1 |
| **Current behavior** | SAN max defaults to 50 (correct for POW 10). If the user sets POW to 15, SAN max stays at 50 instead of becoming 75. |
| **Expected behavior** | On initial creation (before any SAN loss events), SAN max = POW x 5. After the first unnatural encounter, SAN max becomes 99 - Unnatural% (this part already works via `handleUnnaturalChange`). |

**Implementation guidance:**

When POW changes and `unnaturalEncounters` is empty (no unnatural knowledge yet),
recalculate SAN max as `pow * 5`. Once unnatural > 0, the existing
`99 - totalUnnatural` ceiling takes precedence.

```js
const sanMax = totalUnnatural > 0
  ? Math.max(0, 99 - totalUnnatural)
  : pow * 5;
```

Also recalculate BP max on POW change: `BP = currentSAN - newPOW`.

### 6. Weapon table mobile usability

| | |
|---|---|
| **Type** | FRICTION |
| **Priority** | P1 |
| **Files** | `src/components/tabs/CombatTab.jsx` (line 59: `minWidth: 700`) |
| **Problem** | The 10-column weapon table forces horizontal scroll on phones. The ammo ▼/▲ buttons are small touch targets. During combat at the table (phones are common), this is the worst friction point. |

**Implementation guidance:**

Option A (recommended): On viewports < 600px, switch the weapon table to a
**card layout** — one card per weapon showing name, damage, skill%, ammo counter
as stacked rows. The ammo ▼/▲ buttons should be at least 44x44px touch targets
per mobile accessibility guidelines.

Option B: Keep the table but hide less-critical columns (Base Range, Kill Radius,
AP) behind a "show details" toggle on narrow screens.

Use a CSS media query or a `useMediaQuery`-style hook to switch layouts.

### 10. Extract App.jsx state into custom hooks

| | |
|---|---|
| **Type** | DEBT |
| **Priority** | P2 |
| **Files** | `src/App.jsx` (832 lines, 20+ useState calls at lines 24-42) |
| **Problem** | All application state, handlers, drag logic, import logic, and modal management live in a single component. This makes the file hard to navigate and will resist feature additions (e.g., adding the combat quick-ref, dice roller, conditions). |

**Implementation guidance:**

Extract into focused custom hooks:

| Hook | Responsibility | State it owns |
|---|---|---|
| `useCharacterManager` | CRUD, activeId, updateChar, addLogEntry | characters, activeId, loaded |
| `useDragReorder` | Character + weapon drag-and-drop | dragState, weaponDragState |
| `useImport` | PDF + JSON import flow | importState, pendingImport, file refs |
| `useModals` | Modal open/close state | kiaConfirmOpen, confirmDialog, clearLogOpen, sanEventOpen, etc. |

App.jsx should compose these hooks and remain purely a layout/routing component.

## Positive Findings (preserve these)

These are things the app does exceptionally well. Any refactor should preserve
these patterns and behaviors.

### 1. SAN Event Wizard (`SanEventModal.jsx`)

The guided multi-step flow for SAN loss events (category selection, temporary
insanity reactions, repression via WP+Bond, breaking point with disorder picker,
adaptation box tracking, permanent insanity) is mechanically flawless and more
usable than any paper or PDF solution. **Do not simplify this.** It is the app's
killer feature.

### 2. Session Log + Auto-Advancement (`App.jsx:249-277`)

The "check failed skills, hit END SESSION, auto-roll 1d4 per skill" workflow
with full logging is faster than any table method. The session log with
timestamps and typed sources (manual, advancement, san, kia, unnatural, bond)
is an excellent audit trail.

### 3. KIA / Redaction System

Archiving agents with visual black-bar redactions over personal info, preventing
edits via `isLocked`, and allowing "DECLASSIFY" / "REVIVE" is both thematically
perfect and mechanically sound. The full-screen K.I.A. watermark stamp is a nice
atmospheric touch.

### 4. Zero-dependency architecture

React 19 + Vite with zero UI framework dependencies. The entire app is ~2500 lines
of custom code. This keeps the bundle tiny, eliminates upgrade churn, and gives
full control over every interaction. Preserve this philosophy.

### 5. Debounced auto-save with JSON backup

500ms debounced localStorage writes, visible "SAVED" timestamp, JSON export/import,
and PDF import via on-demand PDF.js CDN load. Data persistence is robust and
transparent.
